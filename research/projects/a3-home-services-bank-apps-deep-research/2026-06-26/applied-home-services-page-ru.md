# A3 Home Services: прикладная страница выводов для раздела "Дом"

## Для кого эта страница

Для людей, которым не нужно читать весь research pack, но нужно быстро понять, что можно сделать в банковском разделе `Дом`, какие сценарии имеют смысл первыми, как это может выглядеть в приложении и какие данные A3 должен уметь отдать или маршрутизировать.

## Главный вывод

Раздел `Дом` не должен быть каталогом случайных ЖКХ-сервисов. Его первая полезная версия должна быть рабочей карточкой объекта жилья: адрес, роль пользователя, начисления, оплата, статус поставщика, счетчики, заявки, контакты УК/РСО и понятный путь исправления ошибки.

Ценность A3 здесь не в кнопке "оплатить". Банк уже умеет платить. Ценность A3 в связке `адрес -> объект -> лицевой счет -> начисление -> действие -> внешний статус -> следующий ответственный участник`.

## Как это может выглядеть в разделе "Дом"

### Главный экран объекта

```text
Дом
ул. Ленина, 10, кв. 25
Роль: собственник / плательщик / гость
Источник данных: A3 + поставщик ЖКУ / ГИС ЖКХ / УК
Обновлено: 26.06.2026

[Счет за июнь] 4 820 ₽
Статус: банк может принять платеж -> поставщик еще не квитировал

[Счетчики]
Вода: период открыт до 25 числа
Электричество: показания приняты

[УК и аварийная]
УК: Жилищник района ...
Аварийная: +7 ...

[Заявки]
2 обращения: 1 ожидает ответа УК
```

### Что пользователь должен понять за 10 секунд

| Вопрос пользователя | Ответ интерфейса |
|---|---|
| Это точно мой адрес? | Адрес, помещение, роль, источник сопоставления, дата обновления. |
| За что счет? | Период, услуга, получатель, лицевой счет, сумма, статус задолженности/переплаты. |
| Деньги дошли? | Раздельные статусы: банк принял платеж, поставщик получил, ГИС ЖКХ/поставщик квитировал. |
| Почему счетчик не принимается? | Последнее принятое значение, допустимый период, причина отказа, куда исправлять. |
| Куда звонить при аварии? | Контакт аварийной службы, УК/РСО как исполнитель, граница ответственности банка. |
| Как платить за родителей? | Второй объект, роль плательщика, подтверждение адреса перед оплатой, отдельная история. |

## Прикладные юзкейсы

| Приоритет | Юзкейс | Как выглядит в `Дом` | Данные / рабочий A3-протокол | Потенциальный источник | Что проверить |
|---|---|---|---|---|---|
| P0 | Добавить объект по адресу или лицевому счету | Пустое состояние: "Добавьте дом"; поиск адреса; подтверждение квартиры; роль пользователя | `home.object.resolve`, `home.role.detect`, `home.consent.scope` | ФИАС/адресный справочник, ГИС ЖКХ, данные банка, партнерские данные A3 | Может ли A3 надежно сопоставить адрес, помещение и лицевой счет без лишнего доступа. |
| P0 | Показать счет ЖКУ и получателя | Карточка счета: сумма, период, получатель, лицевой счет, дата данных | `home.accruals.list`, `home.bill.get`, `home.payee.resolve` | ГИС ЖКХ, поставщики ЖКУ, РСО, УК/ТСЖ, платежные документы | Какие поля доступны по API и можно ли показывать источник начисления. |
| P0 | Объяснить статус после оплаты | Timeline: "банк принял" -> "ожидаем поставщика" -> "учтено" | `home.payment.status`, `home.receipt.attach`, `home.supplier.settlementStatus` | Банк, A3 payment layer, поставщик ЖКУ, ГИС ЖКХ/квитирование | Есть ли внешний статус квитирования или только банковский статус операции. |
| P0 | Передать показания счетчиков | Список приборов, период передачи, последнее значение, причина отказа | `home.meters.list`, `home.meter.submit`, `home.meter.rejectionReason` | ГИС ЖКХ, УК/ТСЖ/РСО, данные приборов учета | Ошибки: неверный прибор, закрытый период, значение меньше прошлого. |
| P1 | Найти УК, РСО и аварийную службу | Блок "Кому обращаться"; кнопка звонка; дата актуальности контакта | `home.contacts.get`, `home.emergency.get`, `home.provider.directory` | ГИС ЖКХ, УК/ТСЖ, РСО, региональные справочники, партнерские справочники A3 | Актуальность контактов и юридическая граница ответственности банка. |
| P1 | Создать заявку по проблеме | Тип проблемы, фото, адрес подставлен, статус ответа УК | `home.requests.create`, `home.requests.status`, `home.requests.escalate` | УК/ТСЖ/РСО, ГИС ЖКХ/обращения, партнерские каналы A3 | Может ли банк быть входной точкой, если исполнитель не банк и не A3. |
| P1 | Семейная оплата и несколько объектов | Список объектов: моя квартира, родители, нежилое помещение; роли и напоминания | `home.object.link`, `home.guestAccess.create`, `home.reminders.set` | ГИС ЖКХ/лицевой счет, данные собственника, банковский профиль, consent A3 | Не открыть плательщику лишние данные собственника. |
| P2 | Поверка счетчиков и плановые события | Напоминание о поверке, плановое отключение, календарь дома | `home.events.list`, `home.meter.verificationStatus`, `home.partner.serviceOffer` | Госуслуги.Дом/ГИС ЖКХ, УК/РСО, партнеры поверки | Доступность данных и партнерская экономика. |
| P2 | Паспорт дома и качество дома | Год постройки, серия, этажность, капремонт, управляющая организация | `home.building.profile`, `home.managementCompany.get`, `home.capitalRepair.plan` | ГИС ЖКХ, Фонд капремонта, открытые городские данные, Росреестр/ЕГРН при правовом основании | Доступность, лицензии, персональные данные, актуальность. |

Рабочие названия протоколов выше не являются утверждением, что такие endpoints уже есть у A3. Это список продуктовых контрактов, которые нужно проверить с A3/API discovery.

## Пайплайн данных и продукта

| Шаг | Что делает пользователь | Что должен сделать A3 | Что показывает банк | Если данных нет |
|---|---|---|---|---|
| 1. Подключение | Добавляет адрес или лицевой счет | Сопоставить объект, помещение, роль, источник | "Нашли квартиру, источник данных такой-то" | Ручной ввод лицевого счета, объяснение ограничений. |
| 2. Consent | Подтверждает доступ | Разделить scopes: счета, показания, заявки, история | "Доступ нужен для счетов и показаний" | Минимальный режим "только плательщик". |
| 3. Обзор дома | Открывает карточку `Дом` | Собрать счета, приборы, УК, контакты, статусы | Рабочая панель по объекту | Плашка "данные недоступны, источник не передал". |
| 4. Действие | Платит, передает показания, создает заявку | Отправить действие в правильный контур | Статус действия и доказательство | Показать причину отказа и следующий канал. |
| 5. Внешний статус | Возвращается проверить результат | Получить или запросить статус поставщика/УК | Timeline статусов | Не скрывать, что статус держит внешний участник. |
| 6. Исправление | Спорит с суммой/счетчиком/заявкой | Привязать проблему к документу, прибору, УК/РСО | Кнопка "исправить" с адресатом | Escalation: УК/РСО/ГЖИ/поддержка банка. |

## Что положить в MVP

| MVP-блок | Почему первым | Минимальный состав |
|---|---|---|
| Карточка объекта | Без объекта нельзя безопасно показывать счета и действия | адрес, помещение, роль, источник, дата обновления. |
| Счет ЖКУ | Самый близкий к банку денежный сценарий | сумма, период, получатель, лицевой счет, источник начисления. |
| Раздельный статус оплаты | Главная пользовательская тревога после оплаты | банк принял, поставщик получил, квитирование ожидается/учтено. |
| Счетчики | Регулярное действие с высокой ценой ошибки | прибор, период, последнее значение, отправка, причина отказа. |
| Контакты УК/аварийной | Быстрая бытовая ценность даже без полного transaction layer | УК/ТСЖ/РСО, аварийный телефон, дата актуальности, кто исполнитель. |

Не стоит начинать с полной витрины всех возможных услуг. Лучше сделать 4-5 сильных сценариев, где пользователь понимает адрес, документ, статус и следующего ответственного участника.

## Что спросить у A3 / банка перед PRD

| Вопрос | Зачем нужен |
|---|---|
| Какие объекты A3 может сопоставлять: дом, квартира, помещение, лицевой счет? | Определяет, можно ли делать карточку объекта или только ручной поиск. |
| Какие поля есть по начислениям: сумма, период, получатель, услуга, задолженность, переплата? | Определяет состав карточки счета. |
| Есть ли статус квитирования поставщика или только банковская операция? | Определяет честный timeline после оплаты. |
| Можно ли передавать показания и получать причины отказа? | Определяет, будет ли meters flow полноценным или только ссылкой наружу. |
| Есть ли справочник УК/ТСЖ/РСО и аварийных контактов с датой актуальности? | Определяет полезность аварийного блока. |
| Может ли A3 маршрутизировать заявки и получать статус ответа? | Определяет P1 по обращениям жильцов. |
| Какие consent scopes юридически допустимы для собственника, гостя и плательщика? | Определяет семейный сценарий и privacy. |

## Как измерить, что это полезно

| Метрика | Что она проверяет |
|---|---|
| `object_match_success` | Пользователь нашел правильный адрес/лицевой счет без поддержки. |
| `payment_status_comprehension` | Пользователь отличает банковский статус от статуса поставщика. |
| `post_payment_support_rate` | Меньше обращений "оплатил, но долг висит". |
| `meter_rejection_recovery` | Пользователь после ошибки понимает, что исправить. |
| `time_to_emergency_contact` | В аварийном сценарии контакт находится быстро. |
| `second_object_repeat_payment` | Семейный сценарий дает повторное использование, а не разовый платеж. |

## Короткая рекомендация

Стартовать с сценария `объект -> счет -> оплата -> статус поставщика -> исправление проблемы`. Это самый банковский и самый тревожный путь для жильца. Параллельно готовить `счетчики` и `контакты УК/аварийной`, потому что они превращают раздел `Дом` из платежной витрины в рабочую панель по жилью.

Следующий шаг перед дизайном: взять реальные экраны `Дом` в Т-Банке и Альфа-Банке, провести API discovery с A3 и разметить, какие блоки из этой страницы можно делать как `available`, `manual fallback`, `needs_partner`, `not_available`.
