# Tracing Policy

## Цель

Определить, как фиксировать выполнение агентного workflow для отладки, review и улучшения качества продуктового pipeline.

## Что логировать

- `run_id`
- дата и время запуска;
- краткая цель пользователя;
- выбранный маршрут;
- вызванные agents-as-tools;
- handoffs, если были;
- guardrail decisions;
- approval decisions;
- итоговый `qa_report.verdict`;
- созданные артефакты;
- выполненные проверки.

## Что не логировать

- реальные API keys и токены;
- `.env` значения;
- приватные пользовательские данные без необходимости;
- полные входы/выходы tools, если они содержат sensitive data.

## Sensitive Data Mode

Для production-like запусков:

- `trace_include_sensitive_data=false`;
- tool outputs редактируются перед сохранением;
- внешние источники логируются ссылкой или кратким описанием, а не полным дампом.

## Trace Review

Trace считается пригодным для review, если по нему можно ответить:

- почему выбран такой маршрут;
- какие specialists вызывались;
- какие assumptions были приняты;
- какие guardrails сработали;
- почему QA verdict именно такой;
- какие действия требуют follow-up.
