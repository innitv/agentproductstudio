# Архив: решение по агентному исполнению

## Текущий статус

Активный рабочий сценарий проекта один: **работа через Codex внутри IDE/чата**.

Пользователь пишет запрос Codex, Codex читает `AGENTS.md`, инструкции специалистов, workflow-документы и шаблоны, затем выполняет работу через локальные инструменты, сохраняет артефакты и запускает проверки.

## Что это означает

- Отдельный API-ключ для работы проекта через IDE не нужен.
- Субагенты используются как инструкции и роли внутри процесса через Codex.
- Runtime остаётся вспомогательным техническим слоем для scaffold, validation, doctor, QA, persisted state и внутренних smoke-проверок.
- Рабочая документация не должна предлагать пользователю второй самостоятельный путь исполнения.

## Что было синхронизировано

- `AGENTS.md` описывает один пользовательский режим работы.
- `README.md` описывает проект как Codex agent pack.
- `COMMANDS.md` содержит локальные команды для scaffold, validation, QA, research/reference checks и approvals.
- Инструкции агентов используют единый structured output contract без разделения на пользовательские режимы.
- Шаблоны и workflow-документы описывают единый artifact-driven pipeline.

## Оставшиеся внутренние runtime-файлы

В кодовой базе могут оставаться внутренние helpers и smoke-тесты для проверки старого runtime-слоя. Они не являются пользовательским сценарием и не должны управлять тем, как ведётся работа в проекте.

Допустимый внутренний путь развития: `workflow:start --mode agentic` как approval-gated staged rollout для отдельных specialist stages. Он требует `OPENAI_API_KEY`, target-scoped `model_provider_call` approvals и проверки `workflow:agentic-preflight`; это не заменяет основной сценарий работы через Codex внутри IDE/чата.
