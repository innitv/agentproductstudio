# Директива: Расширение матрицы variants и состояний (component_variants)

## Задача

Дополнить существующий компонент в дизайн-системе недостающими variants, состояниями и/или размерами. На входе — компонент в Figma и его строка в `design/figma/a3-design-system/component-map.md` (в тексте директивы обозначается как `ds/components.md`). На выходе — расширенный Component Set с матрицей `variants x states x sizes`, где все привязки идут через Semantic Variables, и обновлённые `ds/components.md` (актуальная матрица в строке компонента) и `design/figma/a3-design-system/token-map.md` (`ds/foundation.md`) (новые Semantic-токены под состояния, если они потребовались).

Директива работает в двух режимах:
* **`--Single--`** — `component_variants <ComponentName>` для одного конкретного компонента.
* **`--Bulk--`** — `component_variants` без аргумента: агент проходит по `ds/components.md`, находит компоненты с неполной матрицей и предлагает расширить каждый по очереди.

Один компонент = один `use_figma`-вызов в обоих режимах. Bulk просто разбивает работу на серию single-прогонов под капотом.

## Роль

Дизайн-системный архитектор с опытом наращивания матрицы вариантов в существующей ДС. Знает, что write-tools плывут на больших матрицах, поэтому идёт по одному компоненту за прогон. Привычка добавлять новые Semantic-токены через alias к Primitive, не плодить прибитые hex.

> Адаптируй роль под продукт:
> - **B2C веб** -> дизайнер с фокусом на состояния action-элементов (Button, Link), focus-кольца на полях ввода.
> - **B2B SaaS** -> дизайнер с фокусом на состояния в сложных формах, таблицах, dropdown'ах; широкий набор размеров под плотность.
> - **Мобильное приложение** -> дизайнер с фокусом на тач-состояния (pressed вместо hover), accessibility focus.
> - **Лендинг** -> размеры заголовков и кнопок под breakpoints, минимум состояний (default + hover).

## Выбор режима

### Single-режим: `component_variants <ComponentName>`

Используется когда:
- Точечно дорабатываешь один компонент, на котором сейчас собираешь экран.
- Хочешь полный контроль над брифом (точно знаешь, что добавить).
- Демо или обучение.

### Bulk-режим: `component_variants` (без аргумента)

Используется когда:
- ДС большая, нужно одним заходом пройтись по всему UI-киту.
- Заранее не помнишь, у каких компонентов матрица неполная.
- Готовишь UI-кит к этапу сборки экранов (урок 8).

В bulk агент:
1. Читает `ds/components.md`.
2. Считает «неполные» компоненты — у которых нет строки `state:` либо она содержит только `default` и `error`, либо нет строки `size:`, либо variants <= 2.
3. По каждому такому компоненту запускает короткий бриф в чате: *«У Card сейчас 1 variant × 0 состояний. Расширяем?»*

## Детальный пайплайн выполнения (Реконструкция)

### Шаг 3 (Продолжение брифинга в Bulk/Single):
Если пользователь соглашается на расширение:
- Агент предлагает оптимальную сетку расширения на основе роли продукта (например, для B2B SaaS предлагает добавить размеры `s`, `m`, `l` и состояния `hover`, `focus`, `disabled`).
- Согласует финальную матрицу: `[Список размеров] x [Список состояний] x [Список вариантов]`.

### Шаг 4: Генерация Component Set на холсте Figma

Для физического создания/обновления компонентов на холсте агент использует инструмент `use_figma`:
1. **Группировка в Component Set**:
   - В Figma варианты одного компонента объединяются в `COMPONENT_SET`.
   - Имя каждой дочерней ноды (`COMPONENT`) внутри набора должно строго кодировать свойства вариантов через запятую: `size=s, state=hover, checked=false`.
2. **Привязка к токенам**:
   - Цвета заливки, обводки и текста привязываются к семантическим переменным дизайн-системы A3 (например, `--base-accent` для Checked, `--accent-600` для Hover) через RGBA-значения, зафиксированные в `ds/foundation.md`.
3. **Auto Layout**:
   - Все компоненты генерируются с использованием свойств Auto Layout (`layoutMode`, `itemSpacing`, `padding`) для обеспечения адаптивности размеров текста.

### Шаг 5: Обновление локальной документации (Синхронизация)

После успешного выполнения вызова в Figma API агент обязан зафиксировать изменения локально:
1. **Обновление `ds/components.md`** ([component-map.md](file:///c:/Project/product-agent-studio/design/figma/a3-design-system/component-map.md)):
   - В строку/таблицу компонента записывается актуальная обновленная матрица свойств Figma Contract.
   - Обновляется статус компонента на `completed`.
2. **Обновление `ds/foundation.md`** ([token-map.md](file:///c:/Project/product-agent-studio/design/figma/a3-design-system/token-map.md)):
   - Если для нового состояния потребовался уникальный семантический цвет (например, `--a3-input-focus-ring`), он добавляется в таблицу как семантический алиас, ссылающийся на базовый примитивный токен (например, `rgba(83, 151, 235, 0.1)`), **без использования жестко закодированных HEX-значений в коде**.

### Шаг 6: Контроль качества и Guardrails

- **Human-in-the-Loop**: Перед вызовом `use_figma` агент обязан вывести полный сформированный JSON-запрос в чат и получить явное согласие: *«Подтверждаю запись расширенной матрицы для компонента <ComponentName>»*.
- **Запрет на дублирование**: Не создавать новые примитивные цвета. Использовать существующую палитру из `ds/foundation.md`.
- **Проверка доступности (A11y)**: Каждое расширение полей ввода или кнопок обязано включать контрастное состояние `focus` (фокус-кольцо) и состояние `disabled` с читаемым контрастом текста не менее `4.5:1` (или `3:1` для крупных элементов).

### Шаг 7: Локальные триггерные фразы

Агент активируется на следующие интенты пользователя в чате:
- **Single-вызов**: `расширь компонент <ComponentName>`, `добавь варианты для <ComponentName>`, `component_variants <ComponentName>`.
- **Bulk-вызов**: `проверь полноту компонентов`, `запусти аудит матрицы вариантов`, `component_variants bulk`, `расширь матрицу вариантов`.
