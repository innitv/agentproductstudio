# Директива: Стартовая дизайн-система с нуля (ds_baseline)

> [!NOTE]
> Данный документ является локальной политикой проектирования. Глобальный регламент выполнения процесса и шагов генерации зафиксирован в [ds-baseline.workflow.md](file:///c:/Project/product-agent-studio/agent-pack/workflows/ds-baseline.workflow.md) в Agent Pack.

## Задача

Поставить стартовую дизайн-систему на пустой проект: бриф из пяти вопросов -> `design/figma/a3-design-system/token-map.md` (в тексте директивы обозначается как `ds/foundation.md`) с двумя слоями токенов -> две коллекции Figma Variables («Primitive» и «Semantic») -> SectionNode «Foundation» рядом с Wireframes плюс 5-7 базовых компонентов, ссылающихся на смысловые Variables. Один прогон, один файл, один артефакт на выходе.

## Роль

Дизайн-системный архитектор. 5+ лет в продуктовом дизайне, опыт постановки ДС с нуля. Знает разницу между примитивными, смысловыми и компонентными токенами. Умеет писать Variables через Figma API, не только мышкой в панели.

> Адаптируй роль под продукт:
> - **B2C веб** -> дизайнер с фокусом на конверсию и доверие, шкала отступов под плотные карточки.
> - **B2B SaaS** -> дизайнер дашбордов, плотная типографическая шкала, четыре уровня нейтралов.
> - **Мобильное приложение** -> дизайнер с фокусом на тач-таргеты (минимум 44pt), мобильная типографическая шкала.
> - **Лендинг** -> дизайнер маркетинговых страниц, крупная типографическая шкала, акцентный цвет работает на CTA.

## Как устроена директива

Три фазы. Первая — бриф из пяти вопросов. Вторая — `ds/foundation.md` с двумя слоями токенов. Третья — два вызова `use_figma`: создание Variables и отрисовка фундамента плюс базовых компонентов в одном Figma-файле, где уже лежат wireframes.

Жёсткое правило: **никаких MCP-вызовов до апрува текста**. Сначала бриф и foundation в md согласованы, только потом Figma.

**Важно про инструмент:** только `use_figma`. Пишем Variables через Figma Plugin API (`figma.variables.createVariableCollection`, `figma.variables.createVariable`) и рисуем фундамент-секцию обычными `createFrame` / `createRectangle` / `createText`.

**Важно про файл:** работаем в том же Figma Design-файле, где лежат wireframes (если они уже стоят) или планируются финальные экраны. Не плодим файлы — ДС живёт рядом с экранами.

## Фаза 1. Бриф

Задай пять вопросов одним блоком. Никаких уточнений и философии — короткие ответы.
```
1. Отрасль продукта. Одно слово: финтех / e-com / edtech / SaaS / медиа / госуслуги / другое.
2. Аудитория. b2b / b2c / внутренний продукт.
3. Тон. Два прилагательных через запятую: например «строгий, надёжный» или «дружелюбный, простой».
4. Платформа. web / mobile / web+mobile.
5. Локаль. ru / en / ru+en.
```

Сохрани ответы в `design/figma/a3-design-system/brand-brief.md` (`ds/brand-brief.md`). Это якорь решений по тону палитры и типографики на следующих фазах.

> Если пользователь вместо короткого ответа пишет полстраницы — выдели из текста ключевое и переспроси: *«зафиксировал X, верно?»* Не разворачивай бриф в интервью.

## Фаза 2. Матрица генерации Foundation (`ds/foundation.md`) (Восстановленная суть)

На основе 5 ответов брифа агент производит детерминированную трансляцию параметров в конкретные физические и смысловые токены в `ds/foundation.md`.

### 1. Выбор цветовой схемы и шрифта (Тон + Отрасль)

| Отрасль / Тон | Акцентный цвет (Accent) | Нейтральная палитра (Neutral) | Семейство шрифтов (Fonts) |
| :--- | :--- | :--- | :--- |
| **Финтех / SaaS** (строгий, надежный) | **Deep Blue** (`#1998FF`) / **Emerald** (`#00B288`) | **Slate** (Slate 900 для темного, Slate 50 для светлого) | **Inter** или **Outfit** (строгая геометрия) |
| **E-com / Edtech** (дружелюбный, простой) | **Amber/Orange** (`#FA5300`) / **Violet** (`#9924FF`) | **Stone/Zinc** (мягкие теплые серые тона) | **Plus Jakarta Sans** или **Outfit** (округлый тон) |
| **Госуслуги / Медиа** (официальный, чистый) | **Classic Blue** (`#1192BB`) | **Cool Gray** (нейтральный стальной серый) | **Roboto** или **Inter** (высокая читаемость) |

*Для локали `ru` или `ru+en` агент обязан использовать шрифты с полноценной поддержкой кириллицы (Cyrillic-safe: Inter, Outfit, Montserrat).*

### 2. Сетка отступов и плотность (Платформа + Аудитория)

*   **b2b / SaaS / web**:
    - Плотная сетка (Compact/High-density layout).
    - Базовый шаг: `4px` (`space-4`, `space-8`, `space-12`, `space-16`, `space-24`).
    - Скругления: `4px` (`border-radius-s`) для инпутов, `6px` (`border-radius-m`) для кнопок.
*   **b2c / mobile / Лендинг**:
    - Просторная сетка (Comfortable layout).
    - Базовый шаг: `8px` (`space-8`, `space-16`, `space-24`, `space-32`, `space-48`).
    - Скругления: более мягкие, `8px` (`border-radius-l`) и `12px` (`border-radius-xl`).

### 3. Структура `ds/foundation.md`

Документ должен содержать:
1.  **Primitive Collection (Примитивы)**:
    - `primitive.accent.500` = `#HEX`
    - `primitive.neutral.900` = `#HEX` (фон темной темы)
    - `primitive.neutral.50` = `#HEX` (фон светлой темы)
2.  **Semantic Collection (Семантика с алиасами)**:
    - `bg.primary` -> `primitive.neutral.900` (если темная тема)
    - `text.primary` -> `primitive.neutral.50`
    - `accent.cta` -> `primitive.accent.500`
    - `border.subtle` -> `primitive.neutral.700`

## Фаза 3. Запись в Figma (`use_figma`) (Восстановленная суть)

После согласования Markdown-структуры токенов агент совершает два точечных вызова Figma MCP:

### Вызов 1: Figma Variables (Коллекции и Режимы)

Агент создает переменные с привязкой через Figma API:
1.  **Создание коллекции `Primitive`**:
    - Создается группа токенов для цветов (`colors`) и размеров (`numbers` для отступов и радиусов).
2.  **Создание коллекции `Semantic`**:
    - Семантические переменные связываются с примитивами: `bg.primary` ссылается на переменную `primitive.neutral.900`.
3.  **Создание режимов (Modes)**:
    - Если платформа `web+mobile` или аудитория `b2b+b2c`, агент создает два режима в коллекции `Semantic` (например, режим `Light` и режим `Dark` или `Desktop` и `Mobile`), позволяя переключать темы одной кнопкой на фрейме.

### Вызов 2: Отрисовка секции «Foundation» и базовых компонентов

Агент отрисовывает в Figma физический фрейм с именем `Section: Foundation`:
1.  **Визуализация токенов (Color & Typography Swatches)**:
    - Отрисовываются цветные плашки в виде прямоугольников (`RECTANGLE`), к заливке которых привязана соответствующая Figma Variable (`Semantic/bg-primary`, `Semantic/accent-cta`).
    - Отрисовывается шкала шрифтов (`TEXT`) с названиями токенов.
2.  **Генерация базовых 5–7 компонентов**:
    Каждый компонент создается как `COMPONENT` и привязывается строго к созданным смысловым Variables:
    - **Button (Кнопка)**: Auto Layout фрейм. Заливка = `accent.cta`, текст = `text.primary`, скругление = `border-radius-base`, высота = `48px` (для mobile минимальная высота тач-таргета жестко фиксируется в `48px` / `44pt`).
    - **Input (Поле ввода)**: Контейнер с рамкой `border.subtle`, плейсхолдер `text.muted`.
    - **Checkbox (Чекбокс)**: Квадрат со скруглением `4px`.
    - **Switch (Переключатель)**: Подложка со скруглением `full` и внутренний круглый тоггл.
    - **Badge (Бейдж)**: Компактная плашка для статусов.

## Guardrails (Ограничения)

1.  **Сначала Текст, потом Код**: Категорически запрещено делать вызовы в Figma до полного согласования текстовой структуры токенов в `ds/foundation.md` с пользователем в чате.
2.  **Без Дубликатов**: Не использовать жесткие HEX-коды во фреймах компонентов. Все свойства обязаны ссылаться на созданные в первом вызове Figma Variables.
3.  **Локальность**: Все изменения производятся строго в рамках текущего Figma-файла и целевой страницы `agnet` (`node-id=748:247`), чтобы не засорять другие разделы макета.
