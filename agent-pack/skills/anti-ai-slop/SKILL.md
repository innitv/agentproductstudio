# Skill: Anti-AI-Slop Gate (Проверка Содержательности)

## 1. Назначение

Skill применяется перед записью, handoff или внешней публикацией любого содержательного артефакта: research pack, CJM, opportunity roadmap, PRD, copy deck, Notion/Figma export. Он отвечает на один вопрос: **можно ли перенести этот вывод в другой продукт, ничего не потеряв?** Если да — это AI slop, и артефакт получает `needs_revision`.

Нормативный текст — [`agent-pack/workflows/claude-operating-rules.md`](../../workflows/claude-operating-rules.md), раздел 3. Skill делает его исполняемым: даёт порядок проверки, критерии правки и команду lint.

Ключевой принцип: gate проверяет **качество результата, а не список слов**. Запрещённый словарь — только ранний сигнал; артефакт без единого запрещённого слова всё ещё может быть slop.

## 2. Обязательные inputs

- Проверяемый артефакт: `research-summary.md`, `scenario-user-flows.md`, `cjm-map.md`, `opportunity-roadmap.md`, `prd.md`, `copy-deck.md` или publication export.
- Источники, на которые артефакт ссылается (source log, evidence log), — чтобы отличить вывод от догадки.
- `stage-gate-ledger.md` — для записи результата gate.

## 3. Процедура

1. **Lint.** Прогони `yarn research:lint <run-dir>` (или `yarn research:lint <export.md>` точечно). Lint реализует Rules 1-6: не тезисная выжимка, глубина CJM/user-flow, связь roadmap с CJM и валидацией, claims с механизмом, неуниверсальные формулировки, неповторяющиеся строки таблиц.
2. **Ранний сигнал по словарю.** Найди `orchestration`, `rails`, `wedge`, `trust layer`, `seamless`, `unlock`, `north star`, `engine`, `flywheel`, `layer`, `companion`, `playbook`. Если слово не является точным техническим термином в контексте — перепиши. Если термин действительно нужен, рядом должно стоять русское бытовое объяснение.
3. **Тест переносимости.** Для каждого крупного вывода: подставь другой продукт из другой отрасли. Если вывод остался верным — в нём нет доменной детали. Добавь участника, ситуацию, канал, объект оплаты/действия, документ, событие, риск, ограничение рынка или пользовательский страх.
4. **Тест механизма.** «Повысит доверие», «улучшит UX», «снизит friction», «ускорит рост» допустимы **только** если указано: через какой механизм, в каком моменте пути и по какой метрике проверяется. Иначе — вырезать или раскрыть.
5. **Раскрытие метрик через поведение.** `conversion`, `retention`, `activation`, `engagement`, `trust` должны быть описаны как конкретное действие пользователя или бизнеса, а не как абстракция.
6. **Тест разнообразия структуры.** Если разделы или строки таблицы повторяют один шаблон и отличаются только существительными — `needs_revision`.
7. **Тест глубины.** Если пользователь просил проработку, тезисной выжимки недостаточно: после executive summary должны идти подробные кейсы, user flow, CJM или сценарная таблица со связью `персона → ситуация → трение → решение → проверка`. Для CJM обязательны не только stage tables, но и вопрос пользователя на каждом этапе, боль, ответ продукта и метрика.
8. **Тест связности roadmap.** Roadmap и ICE/RICE не живут отдельно от CJM: каждая P0/P1 инициатива ссылается на конкретное трение из CJM и на способ проверки в интервью, прототипе или данных.
9. **Запись.** Результат gate (`passed` / `needs_revision` + список правок) фиксируется в `stage-gate-ledger.md` и `handoff-bundle.md`.

Skill правит формулировки и структуру. Он **не имеет права** добавлять факты, которых нет в источниках: если вывод пустой из-за отсутствия evidence, это `needs_validation`, а не повод дописать убедительную деталь.

## 4. Evidence и failure modes

Evidence: результат `yarn research:lint` (pass/fail + нарушенные правила), список переписанных выводов, запись gate в ledger.

- **`needs_revision`** — lint упал или один из тестов 3-8 провален. Артефакт правится до записи.
- **External write запрещён** при провале lint: Notion/Figma/publication write не выполняется, пока источники, шаблоны или export не исправлены. Это жёсткий gate, а не рекомендация.
- **`needs_validation`** (не failure skill) — вывод пуст, потому что нет источника. Помечается и уходит в claims-to-validate.

Публикационные экспорты и генераторы не должны вставлять hardcoded англоязычные клише; дефолтные контрольные блоки — на русском и объясняют продукт через жизненные сценарии.

## 5. Validation gates

- [ ] `yarn research:lint <run-dir>` пройден или зафиксирован blocker.
- [ ] Тест переносимости пройден: каждый крупный вывод содержит доменную деталь.
- [ ] Каждое обещание эффекта раскрыто через механизм, момент пути и метрику.
- [ ] Структура разнообразна: нет шаблонных строк, отличающихся только существительными.
- [ ] Roadmap/ICE/RICE связаны с конкретными трениями CJM и способом проверки.
- [ ] Результат gate записан в `stage-gate-ledger.md` и `handoff-bundle.md`.
- [ ] При `needs_revision` внешние записи не выполнялись.
