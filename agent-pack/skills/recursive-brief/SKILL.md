# Skill: Recursive Brief (Рекурсивный Бриф)

## 1. Назначение

Skill применяется на `00-intake` — первым, до research и любых артефактов. Он превращает исходный запрос (часто в одну строку) в бриф, по которому можно принимать решения.

Главная ошибка, от которой он защищает: запуск полного pipeline по невыясненному запросу. Стоимость ошибки здесь максимальная — она размножается на все последующие этапы.

Skill **не додумывает** за пользователя: то, что неизвестно, попадает в `Assumptions` или в вопросы, а не в требования.

## 2. Обязательные inputs

- Исходный запрос пользователя (дословно).
- Материалы, которые он приложил: URL, screenshots, документы, существующие артефакты.
- `run-plan.md` — тип работы и маршрут.
- Существующие run/research artifacts по этой же теме, если они есть в репозитории.

## 3. Процедура

### Проход 1 — Expansion (развернуть)

Разверни запрос в полный набор аспектов, даже если пользователь их не назвал:

- Кто пользователь и какую работу он нанимает продукт делать (JTBD).
- Какая поверхность на выходе: `research_report`, `landing`, `product_ui`, `figma_board`, `prototype`, `frontend`, `notion_wiki`.
- Какое главное действие пользователя и как выглядит успех.
- Ограничения: платформа, локаль, бренд, сроки, доступы, комплаенс.
- Что уже существует и может быть переиспользовано (DS, артефакты, код).

### Проход 2 — Deepening (углубить)

Для каждого развёрнутого аспекта задай уточняющий вопрос — и раздели ответы на три категории:

- **Known** — пользователь сказал явно.
- **Assumption** — я предполагаю, и это надо подтвердить.
- **Unknown** — нужно спросить пользователя или проверить в research.

Если `Unknown` блокирует выбор маршрута (тип поверхности, режим DS, наличие reference), **спроси пользователя** через `AskUserQuestion`, а не выбирай молча.

### Проход 3 — Consolidation (свести)

Собери `recursive-brief.md`:

| Раздел | Содержание |
|---|---|
| Запрос | Дословная формулировка пользователя |
| Тип работы | full product / reference-driven / quick draft / limited engineering / cleanup / external write |
| Целевая поверхность | Surface Type Gate |
| Пользователь и JTBD | Кто, что нанимает продукт сделать |
| Главное действие и успех | Что должно произойти |
| Scope | Что входит |
| Non-goals | Что явно не входит |
| Assumptions | Допущения, требующие подтверждения |
| Open questions | Что спросить или проверить в research |
| Существующие источники | Что переиспользуется вместо создания нового |
| Definition of Done | Как поймём, что готово |

Зафиксируй выбранный маршрут в `run-plan.md`.

## 4. Evidence и failure modes

Evidence: `recursive-brief.md` со всеми тремя проходами (не только финальная сводка) и `run-plan.md` с выбранным маршрутом.

- **`blocked`** — критичный `Unknown` не разрешён, а пользователь недоступен: маршрут не выбирается «на глаз».
- **Нарушение** — допущение записано в Scope или Requirements вместо `Assumptions`. Догадка, поданная как требование, — источник самых дорогих ошибок pipeline.
- **`quick draft`** — разрешён только по явной фразе пользователя; в этом режиме бриф сокращается, но результат помечается `partial`/`draft`, а не `success`.

## 5. Validation gates

- [ ] Все три прохода (expansion, deepening, consolidation) выполнены и видны в артефакте.
- [ ] Тип работы и целевая поверхность зафиксированы.
- [ ] Known / Assumption / Unknown разделены; допущения не выданы за требования.
- [ ] Блокирующие вопросы заданы пользователю через `AskUserQuestion`, а не решены молча.
- [ ] Non-goals записаны явно.
- [ ] Существующие источники проинвентаризированы до создания нового.
- [ ] `run-plan.md` содержит выбранный маршрут.
